# Python Feed Linux Pipeline

This repository manages a Linux-focused Python package feed. Packages are scanned, reviewed, published as GitHub Releases, and tracked in `packages.json`.

## Repository Components

- `packages.json`: bootstrap manifest and, after publication, the canonical generated package index
- `README.md`: generated package catalog for consumers
- `bundles/`: generated requirements-style bundle files for multi-artifact installs
- `audit-baselines/`: persisted vulnerability baselines used by the nightly audit
- `scripts/platform_config.py`: shared repo and platform constants
- `scripts/manifest_tools.py`: updates the manifest, bundle files, and generated README
- `scripts/build_review_report.py`: compiles the manual review packet from scan output and package metadata
- `scripts/build_ai_security_review.py`: adds an advisory AI summary when Azure AI Foundry is configured
- `scripts/nightly_rescan.py`: rescans active packages and refreshes vulnerability baselines

## Workflows

1. `package-validation-linux.yml`: validates one package, waits for `PackageApproval`, and optionally publishes artifacts plus manifest changes.
2. `bulk-package-validation.yml`: dispatches one validation run per package in a comma-separated list.
3. `nightly-package-audit.yml`: rescans active packages and refreshes audit baselines.
4. `package-lifecycle.yml`: deprecates, restores, deletes, and optionally purges tracked releases for a package.

## Target Runtime

The feed targets Ubuntu x64 with Python 3.13. The publish stage prefers Python 3.13 Linux x86_64 wheels, then universal wheels, then source distributions.

## Bootstrap Policy

The initial Linux manifest intentionally carries only package names and lifecycle states from the source feed. Release URLs, package files, package types, release tags, bundles, README catalog entries, and audit baselines are regenerated by Linux validation and publish runs.

## Standard Approval Path

1. Run `Package Validation Linux` with `package_version=latest` and `publish_to_feed=false` for dry-run review.
2. Inspect `review_output/approval-report.md`, Snyk outputs, and preview manifest artifacts.
3. Approve or reject through the `PackageApproval` environment.
4. Re-run or dispatch with `publish_to_feed=true` when publication is intended.
5. The publish job creates or updates the release, uploads artifacts, regenerates `packages.json`, `README.md`, and bundle files, then commits the generated state.

## Required Secrets

- `SNYK_TOKEN`: authenticates the Snyk CLI.
- `SNYK_ORG`: selects the Snyk organization.
- `REPO_WRITE_TOKEN`: pushes generated commits and manages releases.
- `AZURE_AI_FOUNDRY_API_KEY`: optional advisory AI review authentication.
- `AZURE_AI_FOUNDRY_ENDPOINT`: optional Azure AI Foundry Responses API endpoint.

## Review Guidance

Approve when the package is needed, compatible with Ubuntu x64 and Python 3.13, has no unresolved critical findings, has acceptable handling for high findings, and has a license posture suitable for internal use.

Reject when the package is incompatible with the target runtime, has unresolved critical vulnerabilities, has unacceptable high-severity risk, has unsuitable license terms, or the workflow artifacts cannot support a confident decision.

Escalate when compatibility, native extension behavior, source-distribution risk, license metadata, release health, or AI review findings need human judgment.

## Verification Checklist

- Dry-run one universal package and one platform-specific package with `publish_to_feed=false`.
- Exercise publish selection for a Linux wheel, a universal wheel, and a source distribution.
- Confirm generated `README.md`, `packages.json`, and `bundles/` do not contain stale release state.
- Run the nightly audit manually against a package after its first Linux release exists.
- Run lifecycle operations on a test package.
- Dispatch a small bulk batch and confirm `PackageApproval` gating is unchanged.
